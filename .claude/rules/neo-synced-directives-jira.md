[NEXUS-WORKSPACE-TYPE-GUARD] En ~/.neo/workspaces.json, los workspaces MIEMBRO de un proyecto federation NUNCA deben tener `"type": "project"`. registry.Add() solo asigna ese tipo cuando `.neo-project/neo.yaml` existe DIRECTAMENTE dentro del workspace path — NO por walk-up desde el directorio padre. Si un miembro tiene `type: "project"` por error: (1) filteredWorkspaces() lo excluye del boot, (2) la API /start devuelve 400 "project-federation roots cannot be started as child processes", (3) OAuth .well-known/oauth-protected-resource devuelve el workspace activo en lugar del solicitado → error "Protected resource does not match expected". Fix: eliminar el campo `type` del entry + kill -HUP $(pgrep neo-nexus). Ejemplo real: strategosia-28463 y strategosia-frontend-71952 bloqueados hasta que se eliminó el campo.
2. [JIRA-WORKFLOW] Doctrina Scrum/Kanban para issues Jira creados via plugin neoanvil: (1) Toda Story tiene parent_key con el Epic. (2) Naming: [<label>] <texto> donde label ∈ {architecture, feature, bug, jira, docs, chore} — coherente con commit scopes del repo. (3) Story flow obligatorio: Backlog → Selected for Development → In Progress → REVIEW → READY TO DEPLOY → Done. Cada transición requiere resolution_comment con señal real (commit hash, tests count, files changed, etc.). (4) Epic flow más corto: Backlog → In Progress (cuando primera child story entra a In Progress) → Done (cuando TODAS las child stories están Done — verificar via get_context, no enforce server-side). (5) Story points Asana scale 1/2/3/5/8/13. Plugin escribe a customfield_10016 + customfield_10038 simultáneamente porque ambos existen en la instancia. (6) Reporter NO está en create screen de instancia your-org.atlassian.net — siempre se setea automáticamente al user del API token; pasar reporter_email da 400. Usar solo assignee_email. (7) Toda story ≥3 SP que cierre cambio sustancial debe atacharle un paquete de docs en ~/.neo/jira-docs/<KEY>/ (README + code/ + images/) — tool jira/attach_artifact pendiente Épica 127.x. Doctrina completa en .claude/rules/jira-workflow.md.

4. [JIRA-WORKFLOW] Para Stories en MCPI, el flujo OBLIGATORIO es: Backlog → Selected for Development → In Progress → REVIEW → READY TO DEPLOY → Done. NUNCA saltar estados, aunque el tablero no lo enforze. Al crear una Story, moverla a "Selected for Development" antes de empezar a implementar (antes de In Progress). Para Epics: Backlog → In Progress → Done (sin intermedios). Error registrado: en sesiones 130.1/130.2, se saltó "Selected for Development" pasando directamente de Backlog a In Progress.
5. [JIRA-EVIDENCE] Después de cada commit que cierra Stories, ejecutar prepare_doc_pack con auto_attach:true para subir evidencias al ticket. Sin esto el ticket queda sin documentación de implementación. Hacerlo ANTES de transicionar a READY TO DEPLOY — el doc-pack es la evidencia que justifica el estado REVIEW→READY. Si neo-hook falla (workspace vacío), llamar manualmente: jira/jira action:prepare_doc_pack ticket_id:MCPI-N commit_hash:<hash> repo_root:<path> auto_attach:true.
6. [MCP-RECONNECT] Tras make rebuild-restart, Claude Code pierde la sesión SSE y las tools MCP retornan "session not found". No existe CLI programático para forzar reconexión (feature gap: github.com/anthropics/claude-code/issues/10129, #30224, #15232). Workaround OBLIGATORIO: usuario debe teclear /mcp en la sesión Claude Code y deshabilitar/habilitar el servidor manualmente. Claude Code tiene auto-reconnect con backoff exponencial (5 intentos, 1-30s) pero solo si detecta la desconexión a tiempo. Mientras esto no esté resuelto: tras rebuild-restart, usar curl directo a :9000 o :9142 para llamadas MCP críticas (jira, certify, radar) en lugar de las tools MCP nativas.
7. [POST-COMMIT-HOOK-WORKSPACE-HEADER] El hook scripts/git-hooks/post-commit DEBE incluir el header X-Neo-Workspace en la llamada curl a Nexus. Sin él, Nexus responde workspace not permitted y prepare_doc_pack falla silenciosamente. Fix (130.3): derivar workspace ID de ~/.neo/workspaces.json buscando path==REPO_ROOT, override via NEO_WORKSPACE_ID env var. Usar bash array para inyectar limpiamente: local curl_args=(...); [[ -n "${WORKSPACE_ID:-}" ]] && curl_args+=(-H "X-Neo-Workspace: $WORKSPACE_ID"). Causa raíz de 26 tickets MCPI sin evidencia.
8. [JIRA-NAMING-DOUBLE-LABEL] Todo título de issue Jira (Epic + Story + Bug) DEBE usar formato `[CATEGORÍA][SCOPE] texto` con AMBOS labels en MAYÚSCULAS. Categorías: FEATURE/BUG/FIX/ARCHITECTURE/CHORE/DOCS/JIRA. Scopes (frontend): UI/PLANIFICADOR/APS/BALANCEADOR/SUPERVISION/HMI/WMS/QUALITY/MAINT/INVENTORY/WORKFORCE/MASTER-DATA/ADMIN/PURCHASING/AUTH/API. Scopes (backend): ENGINE/HANDLER/SERVICE/DTO/REPO/DB/WS/CACHE/WORKER. Scopes (cross): DOCKER/CI/TEST/NEO/DOCS/SHARED. Texto SOLO palabras descriptivas — sin URLs, sin paths, sin "Task X.Y", sin commit hashes. Field labels[] también UPPER: ["CATEGORÍA","SCOPE"]. Antipatrón: lowercase, single-label, paths/URLs en título.
9. [JIRA-DATES-MANDATORY] `create_issue` para Epic Y Story DEBE incluir `start_date` y `due_date` (YYYY-MM-DD) — son obligatorios para roadmap + tracking + Gantt. En tickets retrospectivos (work-already-done), start_date = primer commit relevante, due_date = último commit / merge. El plugin actual NO permite backfill via API — manual UI edit es el único recambio cuando se omiten. Antipatrón: crear ticket sin dates aunque parezca que el flujo "funciona".
10. [JIRA-CHECKBOX-HYGIENE] En el body de la descripción de Story (Criterios de aceptación + Definition of Done), los `[ ]` ARRANCAN VACÍOS al `create_issue`, INCLUSO si el código ya está hecho (ticket retrospectivo). Se flippean a `[x]` durante la lifecycle progression: en Backlog/Selected todos `[ ]`, en In Progress se marcan `[x]` los criterios completándose, en READY TO DEPLOY casi todos `[x]`, en Done todos `[x]`. LIMITACIÓN: el plugin Jira no expone `update_issue` action, entonces el body queda frozen al create. La progression real se documenta en cada `resolution_comment` de transition con bloque "Checkbox progress". Antipatrón: marcar `[x]` al crear el ticket porque el código ya esté escrito — viola el flujo de auditoría.
11. [JIRA-EPIC-STORY-GATING] Story child NO puede salir de Backlog/Selected for Development antes de que el Epic padre esté en `In Progress`. Si el Epic está en Backlog y necesitas avanzar una Story, primero `transition` el Epic. Epic transitions: Backlog → In Progress → Done (sin Selected/REVIEW/READY TO DEPLOY). Epic se mueve a In Progress cuando la primera Story child entra a In Progress. Epic se mueve a Done SOLO cuando TODAS las Stories child están en Done — verificar con `get_context` por cada child antes de transicionar Epic→Done. Antipatrón: avanzar Story sin Epic In Progress, o pasar Epic a Done con cualquier child fuera de Done.
12. [JIRA-DOC-PACK-TIMING] `prepare_doc_pack` se invoca al transitionar la Story a `In Progress` — NO al crear (sin código consolidado todavía), NO al final del lifecycle (perdés evidencia intermedia). Stories ≥ 3 SP que cierran un cambio sustancial DEBEN llevar doc-pack. SP 1-2 opcional pero recomendable para BUGs con audit trail relevante. Pasar `summary` al action para override del README auto-generado con uno explicativo no técnico. `auto_attach: true` zippea + sube en una sola llamada.
13. [JIRA-README-NON-TECHNICAL] El README adjunto en doc-pack debe ser EXPLICATIVO y NO TÉCNICO. SIN mención de "Claude code", "Claude AI", o cualquier herramienta del proceso de desarrollo. Sin paths absolutos, sin commit hashes inline. Centrado en QUÉ cambió y POR QUÉ importa para el usuario final. Estructura recomendada: `## Qué cambió en este sistema` (contexto del usuario) → `## Problema previo` o `## Por qué importa` → `## Qué se corrigió/agregó` → `## Qué cambia para el usuario` (Antes/Ahora) → `## Verificación de que funciona` → `## Archivos involucrados en el cambio` (alto-nivel). Para BUGs agregar `## Cómo se logró`. Pasar el README como `summary` parameter en `prepare_doc_pack` para override del README auto-generado.
