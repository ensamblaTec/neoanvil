# NeoAnvil Synced Directives — DeepSeek Plugin

DeepSeek plugin operational rules: models, caching, session modes, audit discipline.
Extracted from neo-synced-directives.md for size management.
Do not edit manually — content mirrors BoltDB entries.

1. [DEEPSEEK-MODEL-DEFAULT] Default model `deepseek-chat` (V4-flash non-thinking) cubre 80% del trabajo: distill_payload, map_reduce_refactor, generate_boilerplate. Reservar `deepseek-reasoner` (V4-flash thinking) SOLO para red_team_audit y decisiones arquitectónicas — su CoT output multiplica costos 3-5× para misma tarea. Ambos modelos son aliases de deepseek-v4-flash desde 2026-04-26 con pricing unificado: $0.0028/M cache hit, $0.14/M cache miss, $0.28/M output. Context window 1M tokens.
2. [DEEPSEEK-CACHE-DISCIPLINE] El cache hit en DeepSeek V4-flash es 50× más barato que el miss ($0.0028 vs $0.14 per M input tokens). El cache se basa en SHA-256 fingerprint del prefix (Block 1 = system + files concatenados). REGLA: cuando hagas N calls relacionados sobre el mismo set de archivos, mantén `Files[]` IDÉNTICO (mismo orden, misma lista, mismo contenido). Si cambias un archivo entre calls, invalidas el prefix y pagas miss en TODOS los siguientes hasta volver al fingerprint original. Cache window ~1h. Batch tu trabajo: 10 prompts diferentes sobre los mismos 50 archivos = 1 miss + 9 hits = 50× cheaper que 10 misses.
3. [DEEPSEEK-REASONER-CONSTRAINTS] El modo thinking (deepseek-reasoner) tiene restricciones HTTP 400 si pasas: `temperature`, `top_p`, `presence_penalty`, `frequency_penalty`, `logprobs`, `top_logprobs`, `tools` (function calling), o FIM. En multi-turn threaded conversations, NO incluir prior `reasoning_content` en messages — solo el `content` visible (incluirlo dispara 400). El campo `max_tokens` limita TODA la salida incluyendo CoT — un visible response de 500 tokens puede llevar 2000 thinking tokens, así que budget max_tokens accordingly (recomendado: 4096 para audits cortos, 16K-32K para deep reviews).
4. [DEEPSEEK-BABEL-PATTERN] Los prompts a DeepSeek API se envían en INGLÉS. El operator y Claude pueden conversar en español, pero antes de invocar `deepseek/call`, traducir el `target_prompt` a inglés. La razón: DeepSeek fue entrenado primarily en inglés + chino; benchmarks de coding/refactor tasks son notablemente mejores en inglés. Esta regla está marcada en el schema del tool ("Babel Pattern") y debe respetarse incluso para tareas locales (la traducción la hace Claude antes del call, no el plugin).
5. [DEEPSEEK-SESSION-MODE] Ephemeral vs Threaded: distill_payload y map_reduce_refactor son SIEMPRE ephemeral (no requieren continuidad — cada call es self-contained). red_team_audit es threaded (multi-turn audit del mismo código). generate_boilerplate es threaded en background (devuelve task_id, polleas con generate_boilerplate(action:status, task_id:X)). Para threaded: response.thread_id debe persistirse por el caller; siguiente call pasa thread_id explícito. Auto-invalidation: si un archivo en thread.FileDeps cambia (sha256 diff), thread expira y siguiente call retorna error "thread invalidated, start new".
6. [DEEPSEEK-CONTEXT-1M] V4-flash soporta 1M token context window — pueden caber proyectos enteros (~200K LOC). Pero `cache discipline` domina costos: tener 1M de context con cache miss son $140 por call. Mejor pattern: chunkear (ya hace ASTChunker per-function en .go, LineChunker per-N-lines en otros) y batch over fixed Files[] para hit cache 50× cheaper. NO usar 1M context para "throw everything in" — ineficiente. Solo úsalo cuando el problema realmente requiere visibilidad cross-file de un repo entero (ej. encontrar dead code globalmente).
7. [DEEPSEEK-CONFIG-CURRENT] Configuración efectiva del plugin DeepSeek en neo.yaml::deepseek (workspace neoanvil, Épica 143 bootstrap): cache_ttl_seconds=3600 (1h, matches API cache window), max_block1_chars=80000 (~20K tokens prefix cache target), rate_limit_tokens_per_minute=60000, burst=10000, context_pressure_tokens=30000 (auto-distill threshold), session_ttl_minutes=15, session_reaper_interval_minutes=5, chunk_size_tokens=2000, max_tokens_per_session=500000 (= ~$0.28 max budget per session at V4-flash pricing). Modelo default `deepseek-chat` hardcoded en pkg/deepseek/client.go:43; per-action override es follow-up. NO ajustar context_pressure_tokens ni chunk_size_tokens sin medir cache hit rate primero — los defaults están optimizados para cache discipline, no para raw context size.
8. [DEEPSEEK-PRECISION-EMPIRICAL] Datos empíricos sesión PILAR XXVII / 138.A-E (6 rounds × ~106K tokens × ~$0.040 USD): precisión accionable promedio **78-82%** (aplicados + diferidos a deuda válida) sobre 54 hallazgos triados. **Heurística de triage:** descartar cuando DS (1) se contradice en su propio razonamiento (round 2 VULN-002 dijo "incorrecto" y luego "es correcto"); (2) alucina código que no existe (round 3 dijo "cursor pattern" cuando uso `b.Get()` directo); (3) recomienda micro-opt sobre código <100 líneas. Aceptar cuando (a) cita línea+función específica, (b) explica attack vector concreto, (c) sugiere fix incremental sin invasividad.
9. [DEEPSEEK-PROTOCOL-MANDATORY] PROHIBIDO saltar `red_team_audit` argumentando "código mecánico bajo precedente auditado". Confirmado empíricamente: round 5 sobre `pair_audit_events.go` (descrito como "BoltDB persistence repetitivo") arrojó **7/8 hallazgos accionables (87.5% precisión)** incluyendo 2 HIGH bugs reales (EventID collision, severity:0 ambiguity). La heurística "es similar a código ya auditado" subestima sistemáticamente la utilidad de DS. Aplicar DS sobre TODO código nuevo en hot-path independientemente de su parecido superficial con código previo.
10. [DEEPSEEK-PAIR-FEEDBACK-LOOP] PILAR XXVII / 138.E entregó el loop de calibración Pair-mode: tras cada `red_team_audit`, el agent emite `neo_daemon(action:"pair_audit_emit", scope, finding_id, claim_text, severity, files)` por finding. Al certificar archivos, `runPairFeedbackHook` automáticamente intersecta unresolved events con `mutated_files` aprobados, marca resolved con `OutcomeSuccess`, y llama `TrustRecord(pattern, scope, OutcomeSuccess)`. Eventos no resueltos a 30min son barridos por reaper background (state cleanup, no penalty). Tras ~50 audits reales, los trust scores se calibran a tier reflejando comportamiento real del modelo. **Workflow:** después de procesar findings de DS, emitir 1 evento por finding válido al bucket — eso convierte tus aceptaciones+ediciones en datos reales que permitirán activar daemon mode con confianza. Sin emit, los trust scores son sintéticos.
11. [DEEPSEEK-ROUND-PATTERNS] Patrones recurrentes encontrados por DS (rounds 1-6 sobre pkg/state + cmd/neo-mcp): (1) **atomicity** — multi-bucket BoltDB writes secuenciales sin tx única (3 hits); (2) **input validation lookalikes** — `HasPrefix("Aprobado")` permitía "AprobadoFalso"; (3) **clock-resolution collisions** — `unixNano` solo no sobrevive Windows 16ms; (4) **silent-skip masking** — `return nil` en ForEach unmarshal errors oculta corrupción; (5) **omitempty zero-time leaks** — `time.Time{}` serializa "0001-01-01T..."; (6) **pair-mode bypass design tension** — read-only exemptions necesitan map-based dispatch en vez de `if` chains cuando el set crece. Usar como checklist al revisar código BoltDB-heavy o handler-heavy nuevo.
12. [SRE-NEW-CODE-AUDIT-MANDATORY] Toda nueva package que contenga primitivas sensibles (crypto, auth, lock, storage, network) DEBE pasar `deepseek/red_team_audit` con `model:deepseek-v4-pro reasoning_effort:high max_tokens:12000` por archivo crítico ANTES de marcarse "done" en master_plan. Costo: ~$0.005-0.010 por archivo. **Evidencia empírica:** PILAR XXVI Brain Portable shipped ~7000 LOC sin audit; sesión 2026-05-01 detectó 5 SEV 9/8 reales (ChaCha20-Poly1305 nonce reuse en EncryptStream, cross-process Lock TOCTOU en local.go::Lock, plugin thinking_type API 400, keystore.go::Save symlink traversal sin O_NOFOLLOW, IPv4-mapped IPv6 SSRF bypass) que habrían sido CVEs en producción. **Triage rule complementaria:** SEV ≥ 9 findings de DS pro+max DEBEN walkearse mecánicamente (pen-and-paper trace del cipher state / lock state / call graph) antes de aplicar fix — el patrón hallucination dominante es "compose-2-true-premises into false-conclusion" (ej: SEV 10 "salt-swap attack" contra AEAD donde Poly1305 tag-on-nonce defeats the swap). Si el AEAD tag verification step está involucrado y DS no explica cómo pasa, finding probablemente bogus. Triage time presupuestado: 30% del audit time. Catálogo completo de patterns en `~/.claude/projects/.../memory/feedback_deepseek_hallucination_patterns.md`.
13. [DEEPSEEK-OPERATIONAL-EFFICIENCY] Reglas de operación que extraen ~50% más valor del plugin DS por dólar gastado, identificadas en self-audit sesión 2026-05-01 (gasté $0.21; con estas reglas el mismo trabajo costaba ~$0.10). Las 5 reglas son: **(1) Cache discipline cross-call:** estructurar audit batches por subpackage manteniendo `Files[]` como superset estable. Audit `pkg/brain/storage/*` con `Files=[r2.go, local.go, store.go]`; el siguiente call que añada `archive.go` recibe cache hit en los primeros 3 archivos (50× cheaper que miss). Antipattern: cada call con un `Files[]` distinto invalida prefix cache. **(2) Thread continuity per-fix:** `red_team_audit` retorna `thread_id`; persiste el ID y pásalo explícito en la post-fix QA call (`thread_id: "ds_thread_xxx"`, prompt "audit my fix; closed SEV without new regressions?"). DS ya tiene contexto del archivo + findings previos — segundo call cuesta 30% menos en tokens y produce diff-aware analysis. Antipattern: thread fresh cada vez = re-pagar todo el contexto. **(3) Smoke-test antes de calibrar bulk:** UNA call por config para verificar `model` field + `reasoning_tokens` field del response object antes de gastar $0.08 en 36-cell calibration. La sesión 2026-05-01 descubrió post-hoc que `deepseek-chat` ya es thinking-enabled — un smoke-test de $0.005 hubiera evitado la asunción rota. **(4) Usa las 4 actions del plugin, no solo `red_team_audit`:** `distill_payload` resume audit reports de 5KB a 300 tokens (TL;DR para mi context) costo ~$0.001; `map_reduce_refactor` fan-out fixes mecánicos en paralelo cuando el sprint tiene 10+ archivos similares; `generate_boilerplate` produce regression tests desde el finding+fix sin que yo escriba el test a mano. Workflow integrado completo: red_team_audit → distill_payload (resume findings) → mechanical fix → generate_boilerplate (test) → red_team_audit follow-up con thread_id (QA). ~$0.02 por fix vs $0.05+ del workflow actual. **(5) Iterar con DS para finding triage cuando es ambiguo:** después de un audit que retorna 7 findings de severidad mixta, follow-up call con `thread_id` + prompt "Rank these 7 findings by your own certainty. Which 3 are most likely to be REAL based on your reasoning trace?" — DS-self-evaluation tiene mejor accuracy que mi pen-and-paper trace para findings borderline (SEV 6-7). **Aplicación:** este protocolo aplica a TODA sesión que invoque `deepseek/call` más de 3 veces. Para 1-2 calls one-shot el overhead de tracking thread_id no paga. Verificar coverage con tool_stats post-sesión: si `cache_hit_rate < 30%` o `unique_thread_ids > calls / 2`, sesión no siguió el protocolo.
