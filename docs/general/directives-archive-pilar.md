# NeoAnvil Synced Directives — PILAR Implementation Details

Deep implementation notes for PILAR features (federation, CPG, auth, tiers, debt).
Extracted from neo-synced-directives.md to reduce per-file char budget.
Operational quick-rules remain in neo-synced-directives.md.

---

1. [PILAR-XXI-TOOLS] neo_log_analyzer es micro-tool independiente (no intent de neo_radar). Schema: content|log_path + max_lines (default 1000). Analiza: level counts, gaps >1s, top-5 error components (excluye keywords ERROR/WARN/BOOT/INFO), correlación HNSW con corpus incidentes. Usar para analizar archivos INC-*.md directamente o logs de producción. PATTERN_AUDIT solo detecta patrones en INC generados post-Épica 153.C (requieren header **Affected Services:**). INCIDENT_SEARCH requiere IndexIncidents() previo (background al boot). Ambos intents están en neo_radar.

2. [PILAR-XXXI-PROJECT-FEDERATION] Project Federation (PILAR XXXI): crear `.neo-project/neo.yaml` en el directorio raíz del proyecto con campos `project_name`, `member_workspaces` ([]string de paths absolutos), `dominant_lang` (go|python|rust|typescript), `ignore_dirs_add` ([]string, se suma a workspace ignore_dirs). `LoadConfig` descubre el archivo via walk-up desde el workspace y aplica `MergeConfigs` 3-tier: workspace > project > global. `pkg/workspace/registry.Add()` detecta `.neo-project/neo.yaml` y asigna `Type="project"` (vs "workspace"). **BLAST_RADIUS scatter:** `neo_radar(intent:"BLAST_RADIUS", target:"file.go", target_workspace:"project")` → `handleBlastRadiusProjectScatter` — análisis paralelo (sem=4) a todos los member_workspaces vía Nexus POST. Respuesta incluye `project_health_table` (running|error|not_running por workspace). **API Go:** `config.WriteProjectConfig(rootDir, *ProjectConfig)` / `config.LoadProjectConfig(wsDir)` / `config.MergeConfigs(workspace, project)`. Guía completa: `docs/pilar-xxx-xxxiii-guide.md`.

3. [PILAR-XXXII-CPG-FASTBOOT] CPG Fast-Boot (PILAR XXXII): neo-mcp serializa el Code Property Graph a `.neo/db/cpg.bin` (Gob + schema version). Al arrancar: `cpg.LoadCPG(persistPath)` → `cpg.IsCPGStale(workspace, header)` — compara BuildAtUnix vs mtime máximo de todos los .go del workspace. Si no stale: `Manager.LoadSnapshot(g)` → `bootedFast=true`. Si stale o schema mismatch: cold build vía SSA. BRIEFING muestra `boot=fast|cold` en línea CPG. **Campos neo.yaml:** `cpg.persist_path` (default .neo/db/cpg.bin), `cpg.persist_interval_minutes` (default 15, 0=desactivado). **Guardado:** (1) goroutina periódica cada N minutos, (2) SIGTERM handler llama `SaveSnapshot()` antes de shutdown. **Forzar cold build:** `rm .neo/db/cpg.bin`. **Por qué importa:** proyectos grandes con SSA analysis pueden tardar varios segundos al boot — fast-boot elimina ese coste en arranques sucesivos.

4. [PILAR-XXXIII-AUTH-TENANT] Auth Foundation + Multi-Tenant BoltDB (PILAR XXXIII): `~/.neo/credentials.json` (0600) almacena tokens y TenantIDs por provider. `pkg/auth/keystore.go`: `Load(path)` / `Save(creds, path)` / `AddEntry(provider, token, tenantID)` (upsert) / `GetByProvider(provider)`. Al boot main.go: si entry "default" tiene TenantID → inyecta en `cfg.Auth.TenantID` + `state.SetActiveTenant(id)`. `cfg.Auth` es `yaml:"-"` — nunca se persiste en neo.yaml. **Multi-Tenant BoltDB:** cuando TenantID está activo, `memexBucketName()` retorna `"memex_buffer:TenantID"` (lazy CreateBucketIfNotExists en MemexCommit). Sin TenantID → bucket legacy `"memex_buffer"`. Gitignore: `.neo/credentials.json` — NUNCA commitear. Guía: `docs/pilar-xxx-xxxiii-guide.md`. **Migración V7.1:** copiar `~/.neo/credentials.json` a la nueva máquina además de workspaces.json + db/*.db + cpg.bin.

5. [TIER-OWNERSHIP-354.Z] Modelo de knowledge store con 3 tiers deterministas: tier:"nexus" god = Nexus dispatcher (singleton) que posee ~/.neo/shared/db/global.db; children proxy via HTTP REST /api/v1/shared/nexus/*. tier:"project" leader = coordinator_workspace (declarado en .neo-project/neo.yaml); non-coords boot con ks=nil y proxean tier:"project" via Nexus MCP routing al coord. tier:"workspace" = alias de project. bbolt NO soporta mixed RW+RO — Options.ReadOnly:true bloquea en LOCK_SH cuando otro tiene LOCK_EX, por eso leader-only es obligatorio. BRIEFING compact muestra tier:project=leader|proxy:X|legacy. Detalle: docs/tier-ownership.md.

6. [PILAR-LXVI-DEBT] Nexus-level debt registry activo. Cuando BRIEFING compact prefija `⚠️ NEXUS-DEBT:N P0:M |`, ejecutar `neo_debt(action:"affecting_me")` para ver detalles completos. Tras resolver cada issue (típicamente lsof+kill+restart), cerrar con `neo_debt(action:"resolve", scope:"nexus", id, resolution)` — requiere X-Nexus-Token en env NEO_NEXUS_TOKEN si api.auth_token está configurado en nexus.yaml. 4-tier debt: workspace (technical_debt.md local), project (SHARED_DEBT.md), nexus (~/.neo/nexus_debt.md via HTTP al dispatcher), org (reservado PILAR LXVII). Record/resolve MVP implementado solo para workspace/nexus; project/org pendientes. Dedup por sha256(title+sorted_affected_workspaces) dentro de window configurable (default 15min).

8. [PILAR-XXIII-DEBT] Deuda técnica activa en PILAR XXIII (Épicas 157-163): (157) CC>15 en handleSemanticCode/formatFullBriefing/gatherBriefingData/handleProjectDigest/handleASTAudit — mitigar extrayendo helpers privados. (158) INCIDENT_SEARCH retorna 0 si IndexIncidents no corrió al boot (Ollama offline) — usar PATTERN_AUDIT como fallback. (159) NEO_CERTIFY_BYPASS omite registro en heatmap de TECH_DEBT_MAP. (160) AST_AUDIT shadow detection genera FP en loops secuenciales — workaround: renombrar variables. (162) binary_stale_vs_HEAD es solo informativo — aún no bloquea. Prioridad: 162 > 157 > 158 > 159 > 160.

7. [PILAR-LXVII-ORG-TIER] 4-tier federation operacional. Tier "org" habilita coordinación cross-project: `neo_memory(scope:"org")` persiste en `.neo-org/db/org.db` (coordinator_project posee el flock); `neo_memory(learn, scope:"org")` escribe a `.neo-org/DIRECTIVES.md` con ID monotónico + supersedes auto-deprecate; `neo_debt(scope:"org")` lee/escribe `.neo-org/DEBT.md` con AffectedProjects []string y priority P0-P3; 355.B auto-sync mirrorea `.neo-org/knowledge/directives/*.md` a cada workspace miembro como `.claude/rules/org-*.md` al boot (idempotente, source wins conflicts). Non-coordinator projects reciben ErrOrgStoreReadOnly — HTTP proxy pendiente (357.A follow-up: `neo_debt affecting_me` unified 4-tier). Precedencia LLMOverrides inversa (org fills gap donde project+workspace vacíos). Escalation rule: deuda que afecta >1 workspace → scope:"project"; deuda que afecta >1 project → scope:"org"; issues detectados automáticamente por Nexus → scope:"nexus".
